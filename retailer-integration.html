02 Flow A — Händler nutzt alles über FYNDDS

Der ideale Flow: FYNDDS übernimmt das komplette Return-Management. Maximales Rerouting möglich.

Kernprinzip: FYNDDS generiert alle DHL Return-Labels unter dem FYNDDS EKP (Kundennummer). Dadurch ist FYNDDS der rechtliche "Absender" aller Pakete — und kann die Lieferadresse per API ändern, sobald ein Käufer gefunden wird.

Händler

Händler lädt Produkt-Infos hoch: SKU, Beschreibung, Zustand, Originalpreis. FYNDDS berechnet automatisch den Angebotspreis (z.B. 60% vom Original) und veröffentlicht es im Shop.

https://fyndds.com/return/{retailer_slug}

FYNDDS erstellt DHL Return-Label (unter FYNDDS EKP)
Kritisch: Das Label wird im Namen von FYNDDS erstellt, nicht im Namen des Händlers. Absenderadresse = FYNDDS Fulfillment-Adresse. Damit hat FYNDDS die Befugnis, die Lieferadresse später per API zu ändern.

03 Flow B — Händler nutzt 8returns + FYNDDS parallel

Der pragmatische Flow: Händler behält 8returns (weniger Reibung beim Onboarding). FYNDDS hängt sich als Resale-Layer davor.

Das Problem: Wenn 8returns das DHL-Label erstellt, ist der Händler (oder 8returns selbst) der rechtliche Absender — nicht FYNDDS. FYNDDS hat dann keine Befugnis per DHL API die Adresse zu ändern. Lösung: FYNDDS übernimmt die Label-Erstellung auch bei 8returns-Händlern.

Option 1: Webhook-Kopplung (empfohlen)

8returns informiert FYNDDS sobald Return-Request eingeht. FYNDDS erstellt das Label.

  • 8returns bietet Webhook-Konfiguration pro Händler
  • Händler trägt FYNDDS Webhook-URL in 8returns ein
  • Bei jedem Return: 8returns → POST an FYNDDS
  • FYNDDS erstellt DHL-Label (unter FYNDDS EKP)
  • FYNDDS sendet Label per E-Mail an Endkunden
  • 8returns-Dashboard zeigt Tracking (Read-only)

Option 2: 8returns Custom Carrier

Händler konfiguriert FYNDDS als "Custom Carrier" in 8returns — alle Labels kommen von FYNDDS.

  • 8returns unterstützt Custom Carrier via API-Key
  • Händler trägt FYNDDS API-Endpoint als Carrier ein
  • Kein Webhook nötig — 8returns ruft FYNDDS direkt auf
  • Transparenter für Händler (weniger Konfiguration)
  • Einfachste Integration: nur eine Einstellung in 8returns
1
Endkunde des Händlers
Kunde initiiert Retoure im 8returns-Portal
Alles wie gehabt für den Händler. Sein 8returns-Return-Portal ist unverändert. Kunde wählt Produkt + Grund. 8returns verarbeitet den Request.
8returns Portal

8returns sendet Webhook an FYNDDS (oder ruft Custom Carrier API auf)
Statt selbst ein DHL-Label zu erstellen, ruft 8returns den FYNDDS Endpunkt auf und übergibt Produkt- und Kundendaten. FYNDDS antwortet mit Tracking-Nummer + Label-URL.

FYNDDS erstellt DHL-Label (FYNDDS als Absender) + listet Produkt

→ Antwort an 8returns: { "label_url": "https://sidjdgxnjcnornstjolt.supabase.co/storage/v1/object/public/labels/abc.pdf", "tracking_number": "123456789012", "listing_url": "https://fyndds.com/product/xyz" }

Szenario B — Lagerversand: Paket kam schon an → Händler versendet aus eigenem Lager (oder FYNDDS Fulfillment Center) → normaler Versand.

04 Business Case — Warum sollte ein 8returns-Händler zu FYNDDS?

8returns löst Logistik. FYNDDS generiert Umsatz. Beide lösen komplett verschiedene Probleme.

8returns — Was es löst

  • Strukturierter Return-Prozess für Kunden
  • Label-Generierung automatisiert
  • Return-Dashboard für Händler
  • Grund-Tracking der Retouren
  • Kosten: ~€0.50-1.50 pro Return + Abo

FYNDDS — Was es löst

  • Retoure wird zu Umsatz statt zu Kostenpunkt
  • Rerouting spart Lager- & Re-Versandkosten
  • Keine Lagerhaltung für B-Ware nötig
  • Automatische Preisoptimierung
  • Käufer-Netzwerk bereits vorhanden

Konkretes Rechenbeispiel: 100 Retouren / Monat

Ohne FYNDDS (nur 8returns)
Durchschnittlicher Produktwert€80
100 Retouren × €80€8.000 gebundenes Kapital
Lagerkosten (100 Artikel, 3 Monate)- €300
Re-Versand + Verpackung- €600
Wertverlust B-Ware (-30%)- €2.400
8returns Gebühren- €150
Erlös aus B-Ware Verkauf (70% von €80)+ €5.600
Netto-Ergebnis≈ €2.150 Gewinn
Mit FYNDDS (Growth Tier 3%)
100 Retouren × €80 Listing-Preis€8.000 Inventar
50 Rerouting (direkt): kein Lager nötig+ €0 Lagerkosten
FYNDDS Platform Fee (3% × €8.000)- €240
Rerouting Fees (50 × €2.99 — zahlt Käufer)- €0 für Händler
Versand restliche 50 (aus Lager)- €300
Verkaufserlös (100% Abverkauf, 70% Preis)+ €5.600
Schnellerer Abverkauf → kein Wertverlust+ €500 (weniger Markdown)
Netto-Ergebnis≈ €5.560 Gewinn (+158%)

Die 3 stärksten Argumente gegenüber einem 8returns-Händler

Argument8returns allein8returns + FYNDDS
Retoure als Umsatz Retoure = Kostenpunkt. Nur Logistik-Optimierung. Retoure = Umsatz. Jede Retoure wird automatisch gelistet und verkauft.
Rerouting Nicht möglich. Paket muss immer ins Lager. 50%+ der Pakete kommen nie ins Lager. Spart €4-8 pro Paket (Handling + Lager).
Systemwechsel 8returns bleibt unverändert im Einsatz. Kein Wechsel nötig. FYNDDS läuft parallel, 1 Webhook-URL reicht.
Käufer-Netzwerk Händler muss selbst B-Ware vermarkten. FYNDDS bringt fertige Käufer. Keine eigenen Marketingkosten für B-Ware.
Lager-Kapazität B-Ware blockiert Prime-Lagerplatz bis verkauft. Durch Rerouting braucht B-Ware oft gar keinen Lagerplatz.
Zeitaufwand Händler muss Re-Listings, Re-Pricing, Re-Versand managen. Vollautomatisch nach dem einmaligen Setup.

05 retailer-api — Technische Spezifikation

Das ist die fehlende Edge Function die gebaut werden muss. Sie ist der Schlüssel für 8returns-Integration UND für die Händler-App.

POST /functions/v1/retailer-api/returns
Händler oder 8returns meldet eine neue Retoure. FYNDDS erstellt DHL-Label + listet Produkt.
{
  "retailer_id": "uuid",
  "api_key": "sk_ret_...",          // Händler API-Key (generiert beim Onboarding)
  "product": {
    "sku": "SHOE-NIKE-42",
    "name": "Nike Air Max 270",
    "category": "shoes",
    "original_price": 120.00,
    "condition": "new_with_tags",   // new_with_tags | like_new | good | acceptable
    "images": ["https://..."]       // optional: Produktbilder vom Händler
  },
  "customer": {
    "name": "Max Mustermann",
    "email": "max@example.com",
    "phone": "+49 ...",             // optional, für DHL
    "address": {
      "street": "Musterstraße 1",
      "zip": "80331",
      "city": "München",
      "country": "DE"
    }
  },
  "return_reason": "wrong_size"     // optional metadata
}

// Response 200:
{
  "return_id": "uuid",
  "tracking_number": "123456789012",
  "label_url": "https://...pdf",
  "listing_url": "https://fyndds.com/product/xyz",
  "listed_price": 72.00,
  "expected_payout": 69.84          // nach Platform Fee
}
PATCH /functions/v1/retailer-api/returns/{return_id}/status
Händler meldet Status-Update (z.B. Produkt aus Lager versandfertig).
{
  "status": "ready_to_ship" | "shipped" | "cancelled",
  "tracking_number": "...",   // wenn direkt vom Händler versendet
  "notes": "..."
}

Interne Logik der retailer-api

1
API-Key validieren
Händler-ID + API-Key gegen retailers Tabelle prüfen. Bei Fehler: 401.
2
Return in DB anlegen
Insert in returns Tabelle: Status = "label_pending".
3
DHL Return-Label erstellen (dhl-create-label aufrufen)
POST an DHL mit FYNDDS EKP + Kundenadresse als Pickup-Punkt. Label-PDF in Supabase Storage speichern.
4
Produkt auf FYNDDS listen
Insert in products Tabelle. Preis = Original × Händler-Tier-Faktor. Status = "active".
5
E-Mail an Endkunden mit Label
Via Resend: Label-PDF + Tracking-Link + fyndds.com/product/{id} (optional: "Dein Artikel wird weitergegeben"-Message).
Sicherheit: API-Keys haben Format sk_ret_{uuid}. Werden beim Händler-Onboarding (retailer-setup) generiert und in der DB gehasht gespeichert (bcrypt). Nur einmal im Klartext angezeigt — danach nicht mehr lesbar. Rate Limit: 100 Requests/Minute pro Händler.

06 Positioning — Kein Wettbewerb, sondern Erweiterung

FYNDDS verkauft sich nicht als 8returns-Ersatz. FYNDDS verkauft sich als Umsatz-Layer on top.

"Wir ersetzen 8returns"

❌ Hohe Wechselkosten für Händler
❌ Langwieriger Sales-Prozess
❌ Händler muss alles neu konfigurieren
❌ Risiko: "was wenn FYNDDS ausfällt?"

"Wir stecken uns davor — eine URL"

✓ Händler muss nur 1 Webhook-URL in 8returns eintragen
✓ Kein Systemwechsel, keine Schulung
✓ Sofort live: nächste Retoure = erster FYNDDS-Verkauf
✓ ROI messbar ab Tag 1

Sales-Script: Was du einem Händler sagst

"Du nutzt schon 8returns — perfekt, das bleibt genau so. Du trägst einmal unsere Webhook-URL ein. Ab sofort wird jede Retoure die du bekommst automatisch auf unserem Marketplace gelistet. Wenn wir einen Käufer finden bevor dein Paket ankommt, leiten wir es direkt um — du siehst das Paket nie wieder, sparst Lager- und Versandkosten, und bekommst trotzdem bezahlt. Wenn kein Käufer vor Ankunft da ist, läuft alles wie gewohnt. Kein Risiko, keine Kosten bis zum ersten Verkauf."

  • Webhook-URL in 8returns

  • Händler sieht Listing in Dashboard
