    01 — Aktueller Stand

    Wie es jetzt läuft —
    erklärt wie einem Kind

    Was der aktuelle Code macht, was er nicht macht, und wo die Lücken sind.

    Aktueller Geldfluss (Single-Retailer)
      Kunde gibt Kreditkarte ein (z.B. 52,99 €)
          │
          ▼
  Stripe Checkout Session (create-checkout Edge Function)
          │
          ├── Produktpreis:     50,00 €
          ├── Rerouting-Fee:     2,99 €
          │
          ▼
  stripe-webhook feuert → checkout.session.completed
          │
          ├──► 47,50 € → Händler  (via transfer_data.destination)
          ├──►  1,50 € → FYNDDS  (application_fee, 3% von 50€)
          └──►  2,99 € → FYNDDS  (Rerouting bleibt bei FYNDDS)
          │
          ▼
  Käufer bekommt: "Zahlung erfolgreich" E-Mail
           ABER: Keine echte Rechnung. Nur Stripe-Zahlungsbeleg.
    ✅ Was funktioniert
    Zahlung einziehen · Geld an Händler überweisen · Platform-Fee einbehalten · Rerouting-Fee einbehalten · SEPA Debit · Idempotency-Check · Asset-Status auf "sold"
    ❌ Was fehlt
    Echte Rechnungen (§14 UStG) · Multi-Vendor-Checkout · Händler ohne Stripe · Rechnungs-Archivierung (GoBD) · Monatliche Händler-Provision-Rechnung · Partial Refund bei Retoure
    Das Kernproblem in einem Satz: Stripe ist eine Zahlungsmaschine — kein Buchhaltungssystem. Stripe gibt dir einen Beleg (wie ein Kassenbon). Eine §14-UStG-konforme Rechnung mit Händler-USt-ID, fortlaufender Nummer und 10-Jahre-Archivierung musst du selbst bauen oder ansteuern.
    02 — Multi-Vendor Checkout

    3 Produkte → 3 Händler
    in einer Zahlung

    Das aktuelle Problem: Der Code nimmt nur den ersten Händler (primaryAsset.retailer_id) — alle Auszahlungen gehen an Händler A, auch wenn Produkt 2 von Händler B ist.

    Lösung: Separate Charges & Transfers (Industrie-Standard)
      Kunde zahlt 1×  (z.B. 187,97 €)
          │
          ▼
  FYNDDS Stripe-Konto  ← Geld landet KOMPLETT hier
  (KEIN transfer_data.destination beim Checkout)
          │
          ▼
  stripe-webhook: checkout.session.completed
          │
          ├── Loop über alle Assets im Warenkorb
          │
          ├──► Transfer 1: Händler A  →  46,06 €  (49€ - 3% Fee)
          ├──► Transfer 2: Händler B  →  72,75 €  (75€ - 3% Fee)
          ├──► Transfer 3: Händler C  →  58,20 €  (60€ - 3% Fee)
          │
          ├──► FYNDDS behält: 5,49 € Provision (3× Fees)
          └──► FYNDDS behält: 2,99 € Rerouting (1× pro Checkout)
    
    // create-checkout/index.ts — ÄNDERUNG
// VORHER (falsch bei Multi-Vendor):
sessionParams.payment_intent_data = {
  transfer_data: { destination: primaryAsset.retailer_id },  // ❌ nur 1 Händler
}

// NACHHER (korrekt):
sessionParams.payment_intent_data = {
  // KEIN transfer_data — Geld bleibt bei FYNDDS
  metadata: {
    asset_ids: assetIds.join(","),
    order_group_id: crypto.randomUUID(),
  }
}

// stripe-webhook/index.ts — NEU
for (const asset of cartAssets) {
  const fee = calcFee(asset.current_price);
  await stripe.transfers.create({
    amount: Math.round(asset.current_price * 100) - fee,
    currency: "eur",
    destination: asset.retailer.stripe_account_id,
    source_transaction: chargeId,      // koppelt an Original-Charge
    transfer_group: orderGroupId,      // für Reporting
    metadata: { asset_id: asset.id },
  });
}
    DB-Änderung: order_groups Tabelle
    order_groups (neu) → id, buyer_id, total_amount, stripe_payment_intent_id, created_at
    orders → bekommt FK: order_group_id (nullable für Rückwärts-Kompatibilität)
    orders → bekommt: stripe_transfer_id, invoice_pdf_url
    03 — Rechnungen: Die Stripe-Lösung

    on_behalf_of —
    Rechnung kommt vom Händler

    Der entscheidende Trick: Mit on_behalf_of erscheint die Zahlung auf dem Kontoauszug des Käufers unter dem Händler-Namen — nicht unter FYNDDS. Stripe nimmt automatisch die Händler-Daten aus dem Connect-Account.

    Was Stripe Connect beim Onboarding erfasst (= Rechnungsdaten)
      Händler macht Stripe Connect Onboarding:
          │
          ├── ✅ Firmenname          → geht auf Rechnung
          ├── ✅ Geschäftsadresse    → geht auf Rechnung
          ├── ✅ USt-ID / Steuernummer→ geht auf Rechnung
          ├── ✅ IBAN                → für Auszahlung
          └── ✅ Vertretungsberechtigter → KYC-Pflicht

  → Diese Daten sind in Stripe gespeichert.
  → Stripe kann damit automatisch Rechnungen ausstellen.
    
    // create-checkout — on_behalf_of hinzufügen
sessionParams.payment_intent_data = {
  on_behalf_of: retailerProfile.stripe_account_id,  // ← NEU
  transfer_data: {
    destination: retailerProfile.stripe_account_id,
    amount: retailerAmountCents,
  },
  application_fee_amount: platformFeeCents,
}

// Nach dem Checkout: Invoice aus Händler-Account holen
const invoice = await stripe.invoices.retrieveUpcoming(
  { customer: customerId },
  { stripeAccount: retailerProfile.stripe_account_id }  // ← Händler-Account
);
// → PDF enthält Händler-Name, Adresse, USt-ID
    Rechnung 1: Händler → Käufer
    Warenwert inkl. 19% USt · Händler-USt-ID drauf · via Stripe Connected Account · PDF automatisch generiert
    Rechnung 2: FYNDDS → Käufer
    Rerouting-Service 2,99 € · FYNDDS USt-ID · Stripe eigenes Konto · Separater Beleg
    Rechnung 3: FYNDDS → Händler
    Monatliche Provision · B2B Reverse Charge (AT/CH) · Stripe Invoice monatlich gebündelt · Automatisch via Cron
    Was bleibt noch offen: GoBD verlangt fortlaufende Rechnungsnummern (RE-2026-0001, RE-2026-0002...) — Stripe generiert eigene IDs, die nicht sequenziell sind. Lösung: Stripe-PDF in Supabase Storage speichern + eigene laufende Nummer draufstempeln (1 Edge Function, ~4 Stunden Arbeit).
    04 — Händler OHNE Stripe Account

    Was wenn der Händler
    nur Shopify hat?

    Nicht jeder Händler will einen Stripe-Account anlegen. Das ist ein echtes Wachstumsproblem — je einfacher das Onboarding, desto mehr Händler. Hier sind die Lösungen.

    Lösung A: FYNDDS als Merchant of Record (MoR)
    Händler braucht NUR eine IBAN — kein Stripe, kein Login
    Einfachstes Onboarding
    Wie es läuft
      Händler gibt nur an: Firmenname + IBAN + USt-ID
            │
            ▼
  FYNDDS ist rechtlich der Verkäufer (Merchant of Record)
  FYNDDS stellt die Rechnung aus (auf eigenen Namen)
  FYNDDS zahlt Händler per SEPA-Überweisung (wöchentlich)
            │
  Stripe-Setup: Stripe Payouts aus FYNDDS-Konto → Händler-IBAN
  (via Stripe Financial Connections oder manuell)
    Pro
    Händler muss nichts tun außer IBAN angeben · Kein Stripe-Account · Maximale Conversion beim Onboarding · Funktioniert mit Shopify, WooCommerce, SAP, egal
    Con
    FYNDDS haftet als Verkäufer (mehr rechtliche Verantwortung) · FYNDDS braucht eigene USt-Buchführung pro Händler · Komplexere Steuer-Situation bei AT/CH
    ⚠ Wichtig
    Als MoR ist FYNDDS umsatzsteuerrechtlich der Verkäufer — d.h. FYNDDS muss die USt abführen, nicht der Händler. Das erfordert klare AGB und saubere Buchhaltung. Empfehlung: Für AT/CH-Händler separat prüfen lassen.
    Lösung B: Stripe Custom/Express Account — unsichtbar erstellt
    FYNDDS erstellt Stripe-Account IM HINTERGRUND für den Händler
    Mittlerer Aufwand
    Wie es läuft
      Händler gibt an: Firmenname + IBAN + USt-ID + Personalausweis-Scan
            │
            ▼
  FYNDDS erstellt Stripe Custom Account im Hintergrund:
  stripe.accounts.create({ type: "custom", country: "DE", ... })
            │
            ▼
  Händler sieht: NIE ein Stripe-Interface
  Händler sieht: Nur FYNDDS-Dashboard + Auszahlung auf sein IBAN
            │
            ▼
  Intern: Alle Stripe Connect Features verfügbar (Transfers, Invoices)
  FYNDDS übernimmt KYC-Verantwortung → höheres Compliance-Risiko
    Pro
    Händler merkt nichts von Stripe · Volle Rechnungs-Automation · Alle Stripe-Features nutzbar · Dasselbe wie Uber/Lyft für Fahrer
    Con
    FYNDDS übernimmt KYC-Pflicht (AML, Geldwäsche-Check) · Stripe kann FYNDDS-Account einfrieren wenn ein Custom-Account Probleme macht · Mehr Compliance-Aufwand
    Lösung C: Shopify-Händler — Was wirklich passiert
    Shopify ist für den Originalkauf — FYNDDS ist für den Reroute-Verkauf. Kein Konflikt.
    Kein Problem
    Wichtige Erkenntnis: Shopify und FYNDDS überlappen sich NICHT
      Originalkauf: Kunde A kauft bei Händler via Shopify
                → Shopify verarbeitet Zahlung
                → FYNDDS: nicht beteiligt ✅

  Retoure: Kunde A schickt zurück
                → Händler meldet Retoure via FYNDDS-Webhook
                → FYNDDS nimmt Asset auf → Status: "listed"

  Reroute-Verkauf: Kunde B kauft auf FYNDDS Marketplace
                → FYNDDS verarbeitet Zahlung (Stripe, komplett)
                → Händler bekommt Auszahlung von FYNDDS (nicht via Shopify)
                → Shopify weiß davon nichts → kein Problem
    Fazit: Ein Shopify-Händler braucht für den Reroute-Verkauf keinen Shopify-Integration. Er braucht nur: (1) den FYNDDS Return Webhook, (2) eine IBAN oder einen Stripe-Account für die Auszahlung. Shopify und FYNDDS laufen parallel, nicht zusammen.
    Aber: Was wenn Shopify den Rückversand triggert?
    Shopify hat eigene Return-Flows. Wenn der Händler Shopify-Returns nutzt, muss der FYNDDS-Webhook das abfangen BEVOR Shopify die Retoure einbucht. Das geht via Shopify Webhook → FYNDDS API → FYNDDS intercepted das Paket. Timing-kritisch: FYNDDS-Webhook muss vor der Shopify-Lagereinbuchung feuern.

    06 — Potenzielle Gefahren

    Was schief gehen kann —
    Risiko-Matrix

    Alle realen Szenarien die im laufenden Betrieb auftreten können — mit Schweregrad und Gegenmassnahme.

    🔴 KRITISCH
    Händler-Insolvenz während laufender Transaktion
    Händler geht pleite nachdem Käufer gezahlt hat, aber bevor FYNDDS den Transfer ausführt. Oder: Transfer wurde bereits gemacht, Stripe friert Connect-Account ein. Käufer erwartet Ware, Händler existiert nicht mehr.
    ✅ Fix: Transfer erst nach Bestätigung der Zustellung ausführen (aktuell ist das teilweise so). Escrow-Logik: Geld bleibt bei FYNDDS bis Paket bestätigt zugestellt. + Händler-Bonität regelmäßig prüfen (Creditreform API).
    🔴 KRITISCH
    Doppel-Erstattung: Händler erstattet Kunde A + FYNDDS kassiert von Kunde B
    Händler erstattet Originalcustomer A vollständig (wie vorgesehen). Gleichzeitig hat FYNDDS das Produkt an Kunde B verkauft. Händler erhält beide Zahlungen wenn der Code nicht sauber trennt: Originalrückerstattung (Händler→KundeA) und Reroute-Erlös (FYNDDS→Händler) laufen unabhängig.
    ✅ Fix: Asset-Status-Machine strikt halten. Status "rerouted" sperrt jede weitere Händler-Erstattung über FYNDDS. Vertrag mit Händler: Original-Erstattung ist Händler-Sache, FYNDDS zahlt nur den Reroute-Erlös aus.
    🔴 KRITISCH
    Stripe Account Freeze (FYNDDS Platform Account)
    Stripe kann FYNDDS Hauptkonto einfrieren wenn: ungewöhnliche Transaktionsmuster, viele Chargebacks, Compliance-Verstöße, oder Verdacht auf Geldwäsche. Bei einem Freeze: alle Zahlungen gestoppt, Händler-Auszahlungen unmöglich, Kundenzahlungen schlagen fehl.
    ✅ Fix: Früh mit Stripe in Kontakt treten (Stripe Partnership). Klare Geschäftsdokumentation. Chargeback-Rate unter 1% halten. Notfallplan: Backup-Payment-Provider (Adyen als Fallback konfiguriert haben).
    🟠 HOCH
    Gefälschte / gestohlene Ware im Return-Stream
    Händler A wird gehackt. Jemand meldet Retouren für gefälschte Nike-Schuhe. Diese werden auf FYNDDS gelistet. Käufer B kauft "Nike" für 80€ — und bekommt eine Fälschung. FYNDDS haftet als Marktplatz potenziell mit.
    ✅ Fix: Händler-Verifikation beim Onboarding (Handelsregisternummer, Markenvertrag). Nur verifizierte Marken-Händler können Luxus/Sport-Artikel listen. Photo-Verification bei verdächtigen Produkten. AGB: Händler haftet für Echtheit.
    🟠 HOCH
    Chargeback-Betrug durch Käufer
    Käufer B kauft Produkt auf FYNDDS, behauptet danach "Zahlung nicht autorisiert" bei seiner Bank → Chargeback. Geld wird zurückgebucht. FYNDDS verliert Geld + hat das Produkt bereits an Händler ausgezahlt. Bei häufigem Auftreten: Stripe hebt Chargeback-Rate-Alarm.
    ✅ Fix: Stripe Radar (Fraud Detection) aktivieren. 3D Secure für alle Zahlungen erzwingen. Lieferbeweis (DHL Tracking) als Chargeback-Gegenbeweis hinterlegen. Hochrisiko-Kategorien (Elektronik >300€) mit extra Verifizierung.
    🟠 HOCH
    Umsatzsteuer-Probleme bei grenzüberschreitenden Verkäufen (AT/CH)
    Österreichischer Händler verkauft über FYNDDS an deutschen Käufer. Wer führt welche USt ab? CH-Händler ist gar kein EU-Mitglied → Einfuhrumsatzsteuer beim Käufer? Bei falscher Behandlung: Nachzahlungen + Bußgelder.
    ✅ Fix: Für CH-Händler: separate Konditionen, Hinweis auf Einfuhrumsatzsteuer. Für AT: Standard-EU-Regelung, Reverse Charge B2B. Steuerberater für DACH-Varianten konsultieren (einmalig, vor Launch). Taxjar oder Stripe Tax für automatische USt-Berechnung.
    🟠 HOCH
    Timing-Konflikt: Paket unterwegs, Händler hebt Listing auf
    Kunde B kauft. Paket ist noch bei DHL. Händler löscht seinen FYNDDS-Account oder ändert Bankdaten. Transfer schlägt fehl. Oder: Händler-Stripe-Account wird deaktiviert zwischen Kauf und Transfer.
    ✅ Fix: Transfer-Daten beim Kauf einfrieren (snapshot retailer.stripe_account_id in orders-Tabelle). Kontoänderungen erst nach 48h aktiv. Bei fehlgeschlagenem Transfer: automatischer Retry + manuelle Eskalation + Fallback auf SEPA-Payout.
    🟡 MITTEL
    GoBD-Verstoss: Rechnungsnummern nicht fortlaufend
    Das Finanzamt kann bei einer Prüfung die Buchhaltung verwerfen wenn Rechnungsnummern Lücken haben oder nicht sequenziell sind. Stripe generiert eigene IDs (pi_xxx) — diese sind keine GoBD-konformen Rechnungsnummern.
    ✅ Fix: Eigene Rechnungsnummer-Sequenz in DB führen (invoice_sequences Tabelle: year, counter). Bei jeder Zahlung: counter++ → RE-2026-{counter:05}. Diese Nummer auf die Stripe-PDF stempeln oder in eigenem PDF-Template ausgeben.
    🟡 MITTEL
    Händler meldet Retoure doppelt (Race Condition)
    Händler-System sendet Retoure-Webhook zweimal (z.B. bei Timeout-Retry). Produkt wird zweimal in return_assets angelegt. Oder: Gleiche Tracking-Nummer für zwei verschiedene Produkte.
    ✅ Fix: Idempotency-Key auf receive-webhook (tracking_number + retailer_id als UNIQUE constraint in return_assets). Bereits vorhanden im Code für create-checkout — gleiches Muster für receive-webhook einbauen.
    🟡 MITTEL
    Händler ohne vollständige Steuer-Daten ongeboardet
    Händler schließt Onboarding ab ohne USt-ID oder mit falscher Steuernummer. Stripe akzeptiert das zunächst (Daten werden erst bei erster Auszahlung vollständig geprüft). Rechnungen die in der Zwischenzeit ausgestellt wurden sind steuerlich ungültig.
    ✅ Fix: USt-ID-Validierung beim Onboarding erzwingen (EU VAT Validation API — kostenlos, offiziell). Stripe-Account erst aktivieren wenn alle Pflichtfelder inkl. USt-ID validiert. Stripe selbst prüft das teilweise — zusätzliche FYNDDS-Validierung als Sicherheitsnetz.
    🟡 MITTEL
    Geldwäsche via Return-Loop
    Jemand erstellt fake Händler-Account, meldet nie existierende Retouren, kauft das "Produkt" selbst über einen zweiten Account, zieht das Geld via Stripe Connect aus. Klassisches Payment-Laundering-Pattern.
    ✅ Fix: Stripe Connect KYC ist Pflicht (Stripe prüft das). Zusätzlich: Minimum-Zeit zwischen Listing und Kauf (z.B. 24h). Käufer ≠ Händler (E-Mail + IP-Check). Verdächtige Patterns automatisch flaggen (Käufer kauft nur von einem Händler).
    07 — Empfohlener Implementierungsplan

    Was wir wann bauen —
    in der richtigen Reihenfolge

    Priorisiert nach Risiko, Launch-Readiness und Aufwand.

    Phase 1 — Vor Launch (1–2 Wochen)
    1
    on_behalf_of in create-checkout 30 Min. Eine Zeile Code. Sofort: Rechnungen kommen vom Händler, nicht von FYNDDS. Käufer sieht Händler-Name auf Kontoauszug.
    2
    Stripe Invoice PDF → Supabase Storage → E-Mail 1 Tag. Nach checkout.session.completed: PDF abholen, in Storage speichern, per Resend an Käufer senden. Damit haben 80% der Käufer eine ausreichende Rechnung.
    3

    4
    Händler ohne Stripe: IBAN-Only Onboarding (MoR-Modus) 1 Tag. Neues Feld in Händler-Profil: stripe_account_id ODER iban. Wenn nur IBAN: FYNDDS ist MoR, wöchentliche SEPA-Batch-Auszahlung.
    Phase 2 — Post Launch (Monat 1–2)
    5
    Multi-Vendor Checkout: order_groups + separate Transfers 2–3 Tage. Neues DB-Schema + Webhook-Orchestrierung. Wichtig aber nicht Day-1-kritisch, da Cart zunächst auf Single-Retailer beschränkt werden kann.
    6
    GoBD-konforme Rechnungsnummern 4 Stunden. invoice_sequences Tabelle. Sequenzielle Nummer auf PDF stempeln.
    7
    Monatliche Provisions-Rechnung an Händler 1 Tag. Monatlicher Cron: Stripe Invoice per Händler gebündelt. B2B Reverse Charge für AT/CH.
