Was ein Händler tun muss
einmalig, 10 Minuten

03 — Return-Flow

Wie eine Retoure ins System kommt
drei Wege, ein Ergebnis

Eine Retoure kann auf drei Wegen bei FYNDDS landen. In allen Fällen erstellt FYNDDS den DHL-Aufkleber und listet den Artikel.

Weg A — API-Integration
Shop schickt bei jeder Retoure automatisch POST /api/returns mit Artikeldaten. Für Shopify / WooCommerce / Shopware.
Weg B — Dashboard
Händler lädt Artikel manuell im FYNDDS-Dashboard hoch. Foto, Beschreibung, Zustand, Mindestpreis.
Weg C — Bulk CSV
Händler schickt CSV mit vielen Retouren auf einmal. FYNDDS importiert alle, erstellt Labels und listet alles.

05 — Checkout & Zahlung

Wie der Käufer zahlt
Stripe Connect + on_behalf_of

Der Käufer zahlt einmal. Das Geld wird sofort aufgeteilt: Händler bekommt seinen Teil, FYNDDS behält Provision + Rerouting-Gebühr. Die Rechnung kommt vom Händler — nicht von FYNDDS.

Berechnung + Stripe Session
Berechnung:
   Produktsubtotal:       €89.99
   Rerouting-Gebühr:    + €2.99
   ────────────────────────────
   Gesamtbetrag Käufer:  €92.98

   FYNDDS Provision:     min(€15, max(€2.50, €89.99 × 3%)) = €2.70
   Händler bekommt:      €89.99 − €2.70 = €87.29
   FYNDDS behält:        €2.70 + €2.99 = €5.69

Stripe Session:
{
  payment_method_types: ["card", "sepa_debit", "link"],
  payment_intent_data: {
    on_behalf_of: "acct_händler_id",     // Rechnung kommt vom Händler
    transfer_data: { destination: "acct_händler_id", amount: 8729 },
    application_fee_amount: 270          // €2.70 an FYNDDS
  }
}

01 — Das Problem

Was gerade passiert
warum Retouren ein Milliarden-Problem sind

In Deutschland werden jedes Jahr 530 Millionen Pakete zurückgeschickt. Das klingt abstrakt — konkret bedeutet es: Ein Online-Shop mit 100 Bestellungen pro Monat hat durchschnittlich 20-30 Retouren. Jede kostet Geld.

Was passiert heute: Paket kommt zurück → liegt im Lager → verliert jeden Tag Wert → irgendwann wird es mit 50% Rabatt verkauft oder vernichtet. Zwischen Rückkunft und Wiederverkauf vergehen oft 4-8 Wochen. In dieser Zeit verliert der Artikel 20-40% seines Wertes.
Was FYNDDS macht: Das Paket kommt gar nicht erst ins Lager. Es geht direkt vom Rückwegfahrzeug zum nächsten Käufer. Der Händler spart Lagerkosten, Handling und Zeit. Der Käufer bekommt Markenware mit Rabatt. FYNDDS verdient an der Transaktion.
02 — Zwei Wege

Wie Händler zu FYNDDS kommen
mit oder ohne bestehendes System

Es gibt Händler die noch kein spezielles Return-Management-Tool nutzen, und Händler die bereits ein Tool wie 8returns verwenden. FYNDDS funktioniert für beide — ohne dass der Händler sein bestehendes System aufgeben muss.

Flow A — Händler kommt direkt zu FYNDDS
Der Händler hat kein Return-Management-System, oder er möchte wechseln. Er nutzt FYNDDS als komplette Lösung: Return-Portal, DHL-Label, Listing, Verkauf, Payout — alles aus einer Hand.

Vorteil: Maximales Rerouting möglich, weil FYNDDS von Anfang an alle Labels erstellt.
Für wen: Kleine bis mittlere Händler, Shopify/WooCommerce-Shops.
Flow B — Händler nutzt 8returns + FYNDDS parallel
Der Händler nutzt bereits ein Tool wie 8returns und will es nicht wechseln. FYNDDS hängt sich als Resale-Layer davor. Der Händler muss nur eine URL in 8returns eintragen.

Vorteil: Kein Systemwechsel, kein Risiko, sofort live.
Für wen: Händler die bereits Return-Software nutzen.
Was ist 8returns? 8returns (8returns.com) ist ein Return-Management-Tool für Online-Shops. Es bietet ein gebrandetes Return-Portal für Kunden, automatische Label-Generierung und ein Händler-Dashboard. Es löst die Logistik — aber nicht die Frage, wie man die Retoure wieder zu Geld macht. Genau da setzt FYNDDS an.
03 — Flow A

Händler nutzt alles über FYNDDS
der ideale Weg

FYNDDS übernimmt alles: das Return-Portal für den Endkunden, die DHL-Labels, den Marketplace und die Auszahlung. Der Händler hat keinen Mehraufwand — er bekommt einfach am Ende Geld.

1
Händler listet seine Retouren auf FYNDDS
Über das Dashboard oder die API meldet der Händler, welche Produkte als Retoure kommen werden. FYNDDS berechnet automatisch einen Verkaufspreis (z.B. 60% des Originalpreises) und veröffentlicht den Artikel im Shop.
2
Endkunde des Händlers will zurückschicken
Der Kunde des Händlers geht auf eine spezielle Rückgabe-Seite (kann unter der Domain des Händlers laufen, wird aber von FYNDDS betrieben). Er gibt seinen Retourengrund ein und bekommt automatisch einen DHL-Aufkleber per E-Mail zugeschickt.
!
Wichtig: FYNDDS erstellt den DHL-Aufkleber — nicht der Händler
Das ist der entscheidende Punkt. Der DHL-Aufkleber wird im Namen von FYNDDS erstellt (unter FYNDDS' eigenem DHL-Geschäftskonto). Dadurch ist FYNDDS rechtlich der "Absender" des Pakets — und hat die Befugnis, die Lieferadresse später per API zu ändern. Würde der Händler den Aufkleber selbst erstellen, wäre das nicht möglich.
3
Käufer findet den Artikel auf fyndds.com und kauft
Sobald ein Käufer zahlt: Stripe verarbeitet die Zahlung. Das Geld wird sofort in zwei Teile aufgeteilt — Händler-Anteil und FYNDDS-Anteil.
✓
FYNDDS leitet das Paket direkt zum Käufer um
Wenn das Paket noch unterwegs ist (der Händlers-Kunde hat es gerade abgegeben): FYNDDS ändert per DHL API die Lieferadresse. Das Paket fährt direkt zum Käufer. Der Händler sieht das Paket nie wieder. Kein Lager, kein Handling, kein Neuversand.
4
Paket zugestellt → Händler bekommt sein Geld
DHL meldet: Paket zugestellt. FYNDDS überweist automatisch den Händler-Anteil (Verkaufspreis minus Provision). Der Händler bekommt eine E-Mail mit Auszahlungsbestätigung.
04 — Flow B

Händler nutzt 8returns + FYNDDS
kein Systemwechsel nötig

Der Händler behält 8returns genau so wie es ist. Er trägt einmalig eine FYNDDS-URL in 8returns ein — und ab diesem Moment wird jede neue Retoure automatisch auf FYNDDS gelistet und verkauft, wenn möglich sogar umgeleitet.

Das Kernproblem bei dieser Integration: Normalerweise erstellt 8returns die DHL-Labels — mit dem Händler als Absender. Das bedeutet: 8returns (oder der Händler) kontrolliert das Paket, nicht FYNDDS. FYNDDS hat dann keine Befugnis, die Adresse zu ändern.

Die Lösung: 8returns hat eine "Custom Carrier" Funktion. Der Händler trägt dort FYNDDS als seinen Paketdienstleister ein. Ab dann erstellt FYNDDS alle Labels — unter FYNDDS' DHL-Konto. Volles Rerouting möglich.
Option 1 — Webhook-Kopplung (empfohlen)
8returns informiert FYNDDS sobald ein Return-Request eingeht. FYNDDS erstellt das Label.

✓ Händler trägt FYNDDS Webhook-URL in 8returns ein
✓ Bei jedem Return: 8returns → POST an FYNDDS
✓ FYNDDS erstellt DHL-Label (unter FYNDDS EKP)
✓ 8returns-Dashboard zeigt Tracking weiterhin
✓ Volle Rerouting-Befugnis für FYNDDS
Option 2 — 8returns Custom Carrier
Händler konfiguriert FYNDDS als "Custom Carrier" in 8returns — alle Labels kommen von FYNDDS.

✓ 8returns unterstützt Custom Carrier via API-Key
✓ Kein Webhook nötig — 8returns ruft FYNDDS direkt auf
✓ Transparenter für Händler (weniger Konfiguration)
✓ Einfachste Integration: nur eine Einstellung
✓ Volle Rerouting-Befugnis für FYNDDS
1
Einmalig: Händler trägt FYNDDS in 8returns ein
Einmalige Einstellung in 8returns: entweder als "Custom Carrier" (eine URL + API-Key) oder als Webhook-Empfänger. Das dauert 5 Minuten. Danach ändert sich der Workflow des Händlers nicht — 8returns sieht weiterhin alle Retouren im Dashboard.
2
Endkunde gibt Retoure in 8returns ein (wie gewohnt)
Für den Endkunden des Händlers ändert sich gar nichts. Er geht auf dieselbe Return-Seite wie bisher, gibt seinen Grund ein. 8returns verarbeitet den Request — und schickt dabei eine Anfrage an FYNDDS.
!

① DHL-Label unter eigenem EKP (Rerouting möglich)
② Produkt-Listing auf fyndds.com (sofort sichtbar für Käufer)

05 — Warum wechseln / warum parallel nutzen?

Was FYNDDS besser macht
in einfachen Worten

8returns und FYNDDS lösen verschiedene Probleme. 8returns ist ein Logistik-Tool — es organisiert wie Pakete zurückkommen. FYNDDS ist ein Umsatz-Tool — es macht aus Retouren wieder Einnahmen.

8returns allein

  • Strukturierter Return-Prozess für Kunden
  • Automatische Label-Generierung
  • Übersicht über alle Retouren
  • Retoure bleibt ein Kostenpunkt
  • Artikel muss selbst weiterverkauft werden
  • Kein Rerouting — Paket muss ins Lager
  • Lagerkosten + Handling + Neuversand

8returns + FYNDDS

  • Alles von 8returns bleibt erhalten

für Entwickler

Die retailer-api Edge Function ist der Eintrittspunkt für alle externen Systeme — ob 8returns, Shopify Plugin oder direkter API-Call.

POST /functions/v1/retailer-api/returns — Neue Retoure melden
Headers:
  Authorization: Bearer sk_ret_{uuid}    // FYNDDS API-Key des Händlers
  Content-Type: application/json

Body:
{
  "product_name": "Nike Air Max 270",
  "product_sku": "SHOE-NIKE-42",          // optional
  "original_price": 120.00,
  "condition": "new_with_tags",           // new_with_tags | like_new | good | acceptable
  "category": "shoes",
  "customer": {
    "name": "Max Mustermann",
    "email": "max@example.com",
    "address": { "street": "Musterstr. 1", "zip": "80331", "city": "München", "country": "DE" }
  },
  "return_reason": "wrong_size"           // optional
}

Response 200:
{
  "return_id": "uuid",
  "tracking_number": "00340433836536510000",
  "label_url": "https://...pdf",          // DHL-Label zum Ausdrucken
  "listing_url": "https://fyndds.com/product/xyz",
  "listed_price": 72.00,                  // automatisch: 60% des Originalpreises

API-Key Sicherheit: Keys haben das Format sk_ret_{uuid} und werden beim Händler-Onboarding generiert. In der DB wird nur der Hash gespeichert. Rate Limit: 100 Requests/Minute pro Händler. Der Key wird nur einmal im Klartext angezeigt — danach nicht mehr lesbar (wie bei Stripe).

DHL Paket DE: kein API-Rerouting
bestätigt 17. Juni 2026 — neue Strategie definiert

DHL Parcel DE Shipping v2 unterstützt keine Adressänderung während des Transports. Das ist eine technische Grenze von DHL, unabhängig von Kontoart oder Vertrag. FYNDDS hat eine neue Carrier-Strategie definiert.

Option A — Pre-Label (sofort)
Käufer gefunden vor Paketabgabe → FYNDDS erstellt Label direkt an Käufer-Adresse. Kein Carrier-Wechsel, kein Lager. Funktioniert mit DHL. Nächster Schritt: in initiate-rerouting implementieren.
Option B — UPS Intercept (mittelfristig)
Käufer gefunden während Transit → UPS Intercept API ändert Lieferadresse. UPS Geschäftskunden-Account nötig. Macht 24h-Fenster überflüssig. Nächster Schritt: UPS Account beantragen.
Option C — Warehouse (Fallback)
Kein Käufer im Zeitfenster → Paket ins FYNDDS-Warehouse → Re-listing. Für alle Fälle die durch Pre-Label + UPS fallen. Braucht physisches Lager.

Variante A — Standardflow

Ohne 8returns — direktes Return-Portal

Käufer A initiiert die Retoure selbst über das FYNDDS Return-Portal. Das ist der Standardfall ohne externe Retourenplattform. Roter Kasten rechts = Fallback wenn Rerouting scheitert.

Variante B — 8returns Integration

Mit 8returns — automatischer Webhook-Flow

Der Händler nutzt 8returns für seine Retouren. Käufer A macht alles über das 8returns-Portal — FYNDDS wird automatisch via Webhook informiert und erstellt Asset + Label. Ab Schritt 6 (picked_up) identisch mit Variante A.

⚠ retailer-api Edge Function noch nicht implementiert. Schritt 2 (8returns Webhook) ist der Kern dieser Integration. Ohne diese Funktion muss der Händler Retouren manuell über das FYNDDS-Portal einpflegen (Variante A). Die retailer-api muss den 8returns Webhook-Payload parsen und automatisch ein Asset + DHL Label erstellen.
Szenario — Käufer B gibt Paket zurück

Zweite Retoure — Käufer B ist unzufrieden

Käufer B hat das Paket erhalten, will es aber zurückschicken. Dieser Szenario-Flow hat mehrere offene Business-Entscheidungen und ist noch nicht vollständig implementiert.

⚠ Offene Business-Entscheidungen: (1) Wer bezahlt das Rückversand-Label — Käufer B oder FYNDDS? (2) Was ist die Rücksendeadresse — FYNDDS Depot oder direkt Händler? (3) Innerhalb welcher Frist ist Rückgabe möglich? (4) Wieviel wird zurückerstattet — voller Betrag minus Gebühren? (5) Bekommt der Händler seinen Anteil zurück wenn Käufer B reklamiert?
Szenario — Sales Window abgelaufen

Kein Käufer in 48h — was passiert mit dem Paket?

Das 48h Sales-Window endet ohne Kauf. Das Paket wurde schon von DHL abgeholt und ist unterwegs. Es kann nicht mehr umgeleitet werden — es läuft zur ursprünglichen Destination.

Wichtige Frage: Was ist die ursprüngliche Destination des Pakets? Das DHL-Label (Schritt 4, create-dhl-label) muss eine initiale Lieferadresse haben. Das könnte sein: (A) FYNDDS Retourendepot oder (B) direkt der Händler. Diese Entscheidung beeinflusst den gesamten Ablauf bei abgelaufenem Sales-Window.
Szenario — Paket nicht zustellbar

DHL scheitert — Paket kommt zurück

DHL versucht mehrfach zuzustellen (falsche Adresse, Empfänger nicht da) und sendet das Paket zurück. Unterschied: (A) Paket war noch nicht verkauft — kein Refund nötig. (B) Paket war bereits zu Käufer B unterwegs — Refund nötig.

SCHRITT 4 — NICHT IMPLEMENTIERT
DHL Retourenlabel — create-dhl-label
Label muss unter FYNDDS EKP erstellt werden. Ohne eigenes EKP-Label → kein Rerouting möglich.

POST https://api-eu.dhl.com/parcel/de/shipping/v2/orders
billingNumber = FYNDDS_EKP · shipper = Käufer-A-Adresse · consignee = FYNDDS Depot
Antwort: shipmentNumber (Tracking) + label (PDF Base64) → Email an Käufer A
⚠ Das ist die kritischste fehlende Funktion. Ohne sie ist FYNDDS in Produktion nicht launchbar.
SCHRITT 6 — DHL WEBHOOK
DHL: picked_up → Asset gelistet, Sales-Window startet
DHL POST an /functions/v1/tracking-webhook · Header: x-dhl-signature: HMAC-SHA256(body, secret)
1. Signatur validieren — kein Secret gesetzt + keine Signatur → Webhook durchgelassen (Dev-Mode)
2. Asset via tracking_number finden → status="listed", sales_window_end = NOW() + 48h
3. Asset erscheint auf Marketplace
Corner Cases: Doppelter Webhook → idempotent, kein Problem. | Unbekannte Tracking-Number → 200 OK, kein Retry-Storm. | 48h Sales-Window hardcoded — sollte pro Händler konfigurierbar sein.
SCHRITT 9 — RACE CONDITION
create-checkout — zwei Käufer gleichzeitig
Käufer B klickt "Kaufen" → Edge Function prüft: status="listed" AND sales_window_end > NOW()
Dann: status = "reserved" → Stripe Checkout Session → checkoutUrl zurückgeben
⚠ Race Condition: Zwei Käufer gleichzeitig → beide sehen "listed" → beide zahlen → zwei Orders für ein Asset.
Fix: Atomisches UPDATE ... WHERE status='listed' — wenn 0 Rows affected → kein Checkout erstellen.
⚠ Abgebrochener Checkout: Asset bleibt dauerhaft "reserved". Cron-Job nötig: nach X Minuten → zurück auf "listed". Noch nicht implementiert.
SCHRITTE 12–13 — KERNFUNKTION
DHL Rerouting — das Herzstück von FYNDDS
1. DHL Token: POST /parcel/de/account/auth/ropc/v1/token
2. Status check: GET /track/shipments?trackingNumber={nr} → geblockt: out_for_delivery, delivered, returned
3. PATCH: https://api-eu.dhl.com/parcel/de/v1/shipments/{nr}/address
   Body: { consignee: { name1, addressStreet, postalCode, city, country } }
4. DHL 409 → Fallback (Order cancelled + re-listed, kein Refund!) | DHL 2xx → bestätigt → Email Händler
⚠ EKP-Problem: Rerouting nur möglich wenn Label unter FYNDDS EKP. Das ist der Grund warum create-dhl-label so kritisch ist.
⚠ out_for_delivery: Asset wird aktuell bei diesem Webhook NICHT auf "unavailable" gesetzt → Käufer B kann noch kaufen → Rerouting scheitert → Zahlung ohne Lieferung → kein automatischer Refund.
SCHRITT 16 — AUSZAHLUNG
process-payout — Stripe Transfer an Händler
Getriggert durch "delivered" Webhook. stripe.transfers.create({ amount: netCents, destination: "acct_RETAILER" })
netCents = salePriceCents − platformFeeCents − 299 (€2,99 Rerouting Fee) · Platform Fee: 3% hardcoded
⚠ Doppel-Auszahlung: "delivered" Webhook kann mehrfach kommen → Transfer zweimal → Händler erhält doppelten Betrag.
Fix: Unique Constraint auf payouts.order_id + Vorab-Check vor Transfer.
⚠ Multi-Vendor: Bei mehreren Händlern im Warenkorb → kein Transfer pro Händler → Geld steckt im Platform Account.

Offene Punkte — Entscheidungen

Rechnungsstellung, Gebühren & Auszahlung

Antworten auf die 3 zentralen Business-/Tech-Fragen zur Abrechnung zwischen Käufer B, FYNDDS und Händler — mit konkreter technischer Umsetzung.

Frage 1 — Beleg / Rechnung an Käufer B: Wer erstellt ihn im Namen des Händlers?
Entscheidung: Hybrid — Stripe + FYNDDS

Hintergrund: Wenn Käufer B etwas auf FYNDDS kauft, braucht er nach dem Kauf zwei Dinge: (1) eine Zahlungsbestätigung — "deine Zahlung war erfolgreich" — wie ein Kassenbon. Und (2) eine gesetzlich vorgeschriebene Rechnung mit Händlername, Adresse, Steuernummer und ausgewiesenem MwSt-Betrag. Diese Rechnung muss im Namen des Händlers sein, nicht von FYNDDS — denn der Händler ist der eigentliche Verkäufer.

Option a) Stripe — Zahlungsbestätigung (automatisch, kostenlos): Stripe schickt nach jeder erfolgreichen Zahlung automatisch eine E-Mail an Käufer B. Weil FYNDDS die Zahlung technisch "im Namen des Händlers" abwickelt (Stripe-Parameter on_behalf_of), steht auf dieser Bestätigung der Händlername — nicht "FYNDDS". Kein extra Code. Aber: das ist kein steuerrechtlich gültiges Rechnungsdokument — es fehlen Steuernummer, MwSt-Aufschlüsselung und Pflichtangaben nach UStG/UGB.

Option b) FYNDDS generiert die echte Rechnung (Pflicht): Direkt nach dem Kauf (ausgelöst durch Stripe Webhook checkout.session.completed) erstellt FYNDDS eine vollständige PDF-Rechnung — mit Händlername, Adresse, Steuernummer, Artikelbeschreibung, Nettobetrag, MwSt-Satz und Bruttobetrag. Diese PDF wird per E-Mail an Käufer B gesendet (via Resend API) und im Händler-Dashboard hinterlegt.

Option c) Händler schickt selbst (Daten-Connect): Bei Amazon z.B. bekommt der Händler die Bestelldaten und schickt selbst die Rechnung aus seinem eigenen System. Das setzt voraus, dass jeder Händler ein funktionierendes Rechnungssystem hat — nicht realistisch für alle FYNDDS-Händler. Fällt weg.

Ergebnis Hybrid: Stripe liefert sofortige Zahlungsbestätigung (Option a, automatisch), FYNDDS liefert die gesetzeskonforme Rechnung (Option b, eigene Funktion). Käufer B erhält beide per E-Mail.

To-Do: Supabase Edge Function generate-invoice bauen. PDF-Bibliothek: @react-pdf/renderer (Deno-kompatibel) oder externer Dienst (z.B. PDFMonkey). Händlerdaten aus retailers-Tabelle holen, Kaufdaten aus orders-Tabelle. Funktion wird vom stripe-webhook auf checkout.session.completed getriggert.
Frage 2 — FYNDDS Gebühren: Technische Einbehaltung + monatliche Händlerrechnung
Entscheidung: application_fee_amount + Monats-Cron

Einbehaltung (automatisch per Transaktion): Das Feld application_fee_amount in der Stripe Checkout Session enthält Platform Fee + Rerouting Fee. Stripe zieht diesen Betrag automatisch ab und überweist ihn an den FYNDDS Platform Account. Kein manueller Eingriff nötig — läuft per Transaktion.

Monatliche Sammelrechnung je Händler:
Cron-Job am 1. des Monats → stripe.applicationFees.list({created: {gte: Monatsbeginn, lt: Monatsende}}) → nach stripe_account_id gruppieren → pro Händler eine PDF-Rechnung (Transaktionsliste, Platform Fee %, Rerouting Fees, Total) → Versand via Resend API.

Stripe API: GET /v1/application_fees mit Zeitfilter. Händler erhält eine Rechnung pro Monat für alle Transaktionen — keine Einzelabrechnung pro Verkauf.

To-Do: Edge Function generate-monthly-invoice + Supabase pg_cron Schedule (oder GitHub Actions Cron). Daten-Aggregation aus Stripe Application Fees API. Rechnung in Supabase Storage ablegen + Link im Händler-Dashboard anzeigen.
Frage 3 — Händler-Auszahlung: Wer überweist den Händleranteil?
Entscheidung: Stripe ✓ (Favorit)

Option a) Stripe — empfohlen: FYNDDS ruft nach DHL-Webhook delivered die Edge Function process-payout auf → stripe.transfers.create({amount, currency: "eur", destination: händler_stripe_account_id}) → Geld wandert aus FYNDDS Platform Balance direkt auf den Stripe Connect Express Account des Händlers → von dort Auto-Payout auf seine Bankverbindung (Standard 2–3 Werktage, kostenlos).

Stripe-Zusatzgebühren: Transfers zwischen Connect-Accounts (Platform → Connected) sind kostenfrei. Stripe Processing Fee (~1,4% + 0,25€) fällt ohnehin an. Instant Payout (sofort auf Bankkonto): +1,0% — optional, auf Wunsch des Händlers, zu dessen Lasten. FYNDDS deckt normale Stripe-Gebühren aus der Platform Fee (bereits einkalkuliert).

Option b) FYNDDS als Zwischenstation: FYNDDS hält Kundengeld → in DE regulatorisch problematisch (Zahlungsdienste-Aufsichtsgesetz, BaFin-Lizenz nötig). Unnötiger Aufwand. Entfällt.

✓ Bereits teilweise implementiert: process-payout Edge Function existiert. Fehlend: Trigger nach delivered Webhook korrekt verknüpfen + Fehlerbehandlung wenn Transfer fehlschlägt (Retry-Logik, Alert an FYNDDS Ops).
