01 — Gesamtüberblick

Der komplette Flow
von Retoure bis Auszahlung

FYNDDS fängt Retouren ab bevor sie ins Lager kommen. Dafür braucht es drei APIs: DHL für die Paketumleitung, Stripe Connect für den Geldfluss, Resend für E-Mails.

Master Flow — Überblick
HÄNDLER                    FYNDDS                     KÄUFER
    │                          │                              │
    │  1. Retoure eingeleitet   │                              │
    ├─────────────────────────►│                              │
    │                          │  2. DHL Retourenaufkleber    │
    │◄─────────────────────────┤     erstellen (FYNDDS Konto) │
    │                          │                              │
    │  3. Artikel auf          │                              │
    │     Marketplace listen   │                              │
    │                          ├──────────────────────────────►
    │                          │  4. Käufer kauft + zahlt     │
    │                          │◄─────────────────────────────┤
    │                          │                              │
    │  5. DHL Webhook:         │                              │
    │     Paket gescannt       │                              │
    │     → in Netz von DHL    │                              │
    │                          │                              │
    │                          │  6. DHL API: Adresse ändern  │
    │                          ├─────────────────►  DHL       │
    │                          │◄─────────────────  DHL       │

Kernprinzip: FYNDDS erstellt den DHL-Retourenaufkleber unter dem eigenen DHL-Geschäftskonto. Das gibt FYNDDS die Kontrolle über das Paket und erlaubt die Adressänderung via API — ohne dass der Händler irgendetwas tun muss.
02 — Händler-Onboarding

Was ein Händler tun muss
einmalig, 10 Minuten

Jeder Händler braucht einen Stripe Connect Express Account. Das ist Pflicht — kein Stripe Connect, kein Listing. Die Einrichtung läuft vollautomatisch über unsere retailer-setup Edge Function.

1
Händler registriert sich auf fyndds.com
E-Mail, Firmenname, USt-ID, Adresse. Supabase Auth erstellt den User. Tabelle profiles bekommt role = 'retailer'.
2
Stripe Connect Express Onboarding
Edge Function retailer-setup wird aufgerufen.
Stripe API POST /v1/accounts → type: "express"
Stripe API POST /v1/account_links → Onboarding-URL generieren
Händler wird zu Stripe weitergeleitet: Bankdaten, Personalausweis, USt-ID.
Stripe verifiziert alles — FYNDDS muss nichts manuell prüfen.
Webhook account.updated → stripe_onboarding_completed = true in DB.
3
API-Key für Shop-Integration
Händler bekommt einen FYNDDS API-Key (UUID in retailer_api_keys Tabelle).
Damit kann der Händler Retouren per API melden (oder manuell im Dashboard).
Shop-Plugins für Shopify / WooCommerce / Shopware nutzen diesen Key.
4
Fertig — Händler kann Retouren listen
✓ Stripe Connect aktiv → Geldfluss möglich
✓ DHL-Aufkleber werden automatisch durch FYNDDS erstellt
✓ Artikel erscheinen sofort auf dem Marketplace
Wichtig: Ohne stripe_onboarding_completed = true kann kein Checkout gestartet werden. Die create-checkout Edge Function prüft das als erstes.
03 — Return-Flow

Wie eine Retoure ins System kommt
drei Wege, ein Ergebnis

Eine Retoure kann auf drei Wegen bei FYNDDS landen. In allen Fällen erstellt FYNDDS den DHL-Aufkleber und listet den Artikel auf dem Marketplace.

Weg A — API-Integration
Händler-Shop (Shopify/WooCommerce) schickt bei jeder Retoure automatisch:
POST /api/returns
mit Artikeldaten, Originalpreis, Zustand, Fotos → FYNDDS-System übernimmt alles.
Weg B — Dashboard Upload
Händler loggt sich ins FYNDDS-Dashboard ein, lädt Artikel manuell hoch: Foto, Beschreibung, Zustand, Mindestpreis. Gut für kleine Händler ohne Tech-Team.
Weg C — Bulk Import
Händler schickt CSV mit vielen Retouren auf einmal. FYNDDS importiert, erstellt alle Labels und listet alle Artikel in einem Batch.
Return-Eingang → Asset-Erstellung
Händler API-Call:
POST /api/returns
{
  "retailer_api_key": "abc123",
  "items": [{
    "product_name": "Nike Air Max 270",
    "original_price": 149.99,
    "condition": "new_with_tags",         // new_with_tags | like_new | good
    "category": "shoes",
    "size": "42",
    "photos": ["https://...jpg"],
    "customer_return_address": {
      "street": "Musterstraße 1",
      "city": "Berlin",
      "postal_code": "10115",
      "country": "DE"
    }
  }]
}

FYNDDS Response:
{
  "asset_id": "uuid",
  "dhl_tracking_number": "00340433836536510000",
  "return_label_url": "https://...pdf",     // DHL Label zum Ausdrucken
  "marketplace_url": "https://fyndds.com/p/uuid",
  "listed_price": 89.99                     // auto-calculated: 60% of original
}
Automatische Preisberechnung: Startpreis = Originalpreis × 0.6. Der Preis fällt automatisch alle 24h um 5% wenn kein Kauf — bis zu einem Mindestpreis (vom Händler festgelegt oder 30% des Originalpreises).
04 — DHL API

Rerouting technisch erklärt
exakt was passiert

Das ist der technische Kern von FYNDDS. FYNDDS erstellt alle Retourenaufkleber unter dem eigenen DHL-Geschäftskonto (EKP-Nummer). Das gibt FYNDDS die Autorität, Pakete während des Transports umzuleiten.

DHL APIs die wir nutzen

APIWofürEndpoint
DHL Parcel DE — Shipment API Retourenlabel erstellen POST /parcel/de/v1/shipments
DHL Tracking API Paketstatus abfragen GET /track/shipments?trackingNumber=
DHL Webhook / Push Notifications Echtzeit-Events empfangen Eingehend: POST /receive-webhook
DHL Address Change Empfänger im Transit ändern PATCH /parcel/de/v1/shipments/{id}/address

Schritt 2: DHL Webhook — Paket gescannt

DHL Push Notification — was wir empfangen
DHL schickt bei jedem Scan-Event einen POST an unsere Edge Function receive-webhook.
Die wichtigsten Events:

DELIVERED_TO_SENDER — Übergabe an DHL durch Kunden ✓ Rerouting möglich
IN_TRANSIT — Paket im Netz unterwegs Rerouting möglich
OUT_FOR_DELIVERY — Auf dem Zustellfahrzeug zu spät
DELIVERED — Zugestellt → Payout triggern
EXCEPTION — Problem (beschädigt, nicht abgeholt etc.)

Schritt 3: Adresse ändern (Das Herzstück)

DHL Address Change — Rerouting API Call
// Nur möglich wenn: Status = IN_TRANSIT oder DELIVERED_TO_SENDER
// NICHT möglich wenn: OUT_FOR_DELIVERY oder DELIVERED

PATCH https://api-eu.dhl.com/parcel/de/v1/shipments/00340433836536510000/address
Authorization: Basic {base64(username:password)}
Content-Type: application/json

{
  "consignee": {
    "name1": "Max Mustermann",          // ← Käufer-Name
    "addressStreet": "Kauferstraße 5",
    "postalCode": "80331",
    "city": "München",
    "country": "DEU"
  },
  "comment": "FYNDDS Rerouting - Order #uuid"
}

Response 200 OK:
{ "result": "address_changed", "newTrackingNumber": "003404..." }

Response 409 CONFLICT:
{ "error": "OUT_FOR_DELIVERY", "message": "Too late for redirect" }
→ Fallback: Paket kommt zu FYNDDS Fulfillment Center → wird von dort neu versendet
Kritischer Zeitpunkt: Das Rerouting muss BEVOR das Paket auf den Zustellfahrzeug geladen wird. Sobald DHL OUT_FOR_DELIVERY meldet, ist es zu spät. Deshalb prüfen wir bei JEDEM IN_TRANSIT Event: gibt es einen Käufer?

Fallback-Szenarien

✓ Idealfall: Käufer da, Paket in Transit
DHL API → Adresse ändern → direkt zum Käufer → Payout nach Zustellung.
⚡ Kein Käufer bis OUT_FOR_DELIVERY
Paket geht zu FYNDDS Fulfillment-Adresse. Dort einlagern bis Käufer da. Dann neu versenden (normaler DHL-Versand). Listing bleibt aktiv.
⚡ Käufer da, aber zu spät für Redirect
Paket kommt zu FYNDDS Fulfillment. Neues DHL-Label zum Käufer erstellen. Rerouting-Gebühr trotzdem fällig (Mehraufwand wird intern abgewickelt).
✗ Kein Käufer, kein Platz im Fulfillment
Paket geht an Händler zurück (Standard-Retoure). FYNDDS berechnet nichts. Händler zahlt nur wenn Verkauf erfolgt.
05 — Checkout & Zahlung

Wie der Käufer zahlt
Stripe Connect + on_behalf_of

Der Käufer zahlt einmal. Das Geld wird sofort aufgeteilt: Händler bekommt seinen Teil, FYNDDS behält Provision + Rerouting-Gebühr. Dank on_behalf_of kommt die Rechnung vom Händler — nicht von FYNDDS.

Stripe Checkout Session erstellen — create-checkout Edge Function
1. Berechnung:
   Produktsubtotal:     €89.99
   Rerouting-Gebühr:    + €2.99
   ─────────────────────────────
   Gesamtbetrag Käufer: €92.98

   FYNDDS Provision:    min(€15, max(€2.50, €89.99 × 3%)) = €2.70
   Händler bekommt:     €89.99 - €2.70 = €87.29
   FYNDDS behält:       €2.70 (Provision) + €2.99 (Rerouting) = €5.69

2. Stripe Session:
stripe.checkout.sessions.create({
  line_items: [
    { price_data: { currency:"eur", product_data:{name:"Nike Air Max 270"},
                    unit_amount: 8999 }, quantity: 1 },
    { price_data: { currency:"eur", product_data:{name:"Rerouting-Gebühr"},
                    unit_amount: 299 }, quantity: 1 }
  ],
  mode: "payment",
  payment_method_types: ["card", "sepa_debit", "link"],
  payment_intent_data: {
    on_behalf_of: "acct_händler_connect_id",    // Rechnung vom Händler
    transfer_data: {
      destination: "acct_händler_connect_id",
      amount: 8729                               // €87.29 → Händler
    },
    application_fee_amount: 270                  // €2.70 → FYNDDS
  },
  metadata: {
    asset_id: "uuid",
    retailer_id: "uuid",
    rerouting_fee: 299
  }
})

3. Was passiert dann:
   Käufer zahlt → stripe-webhook: payment_intent.succeeded
   → Asset Status: "sold"
   → Order erstellen in DB
   → initiate-rerouting Edge Function aufrufen
06 — Rerouting Logic

Was genau passiert nach dem Kauf
der komplette Entscheidungsbaum

Die initiate-rerouting Edge Function ist das Herzstück von FYNDDS. Sie prüft den Paketstatus und entscheidet welchen Weg das Paket nimmt.

initiate-rerouting — Entscheidungsbaum
payment_intent.succeeded
       │
       ▼
Fetch Order + Asset + DHL Tracking Status
       │
       ├── Status = IN_TRANSIT oder DELIVERED_TO_SENDER?
       │       │
       │       ▼ JA
       │   DHL PATCH /shipments/{id}/address
       │   → Käufer-Adresse setzen
       │       │
       │       ├── 200 OK → Order.status = "rerouting_confirmed"
       │       │          → E-Mail an Käufer: "Paket ist unterwegs zu dir"
       │       │          → E-Mail an Händler: "Verkauf bestätigt"
       │       │
       │       └── 409 Conflict (zu spät)
       │                  → Order.status = "fulfillment_required"
       │                  → Paket geht zu FYNDDS Fulfillment
       │                  → fulfillment Queue erstellen
       │
       ├── Status = NOT_YET_SHIPPED?
       │       │
       │       ▼ Paket noch nicht abgegeben
       │   Warten auf DHL DELIVERED_TO_SENDER Event
       │   → Rerouting-Job in Queue mit Käufer-Adresse speichern
       │   → Wenn Event kommt: sofort DHL PATCH ausführen
       │
       └── Status = DELIVERED (zu FYNDDS Fulfillment)?
               │
               ▼
           Fulfillment-Flow:
           → Neues DHL Label zum Käufer erstellen
           → Normaler Versand
           → Tracking-Nummer an Käufer
07 — Payout

Wie der Händler sein Geld bekommt
automatisch nach Zustellung

Der Payout ist vollautomatisch. Er wird durch den DHL "DELIVERED" Webhook ausgelöst. Das Geld liegt bereits bei Stripe — es wird nur freigegeben.

1
DHL Webhook: DELIVERED Event
DHL sendet POST /receive-webhook mit Event DELIVERED und Tracking-Nummer.
Edge Function findet zugehörige Order in DB.
2
process-payout wird aufgerufen
Order Status → delivered
Stripe Transfer wird ausgeführt (war schon durch transfer_data im Checkout vorbereitet).
Stripe API POST /v1/transfers → €87.29 an Händler Connect Account
FYNDDS behält €5.69 (Provision €2.70 + Rerouting €2.99) automatisch.
3
E-Mails via Resend
Resend API → Käufer: Zustellbestätigung + Rechnung
Resend API → Händler: Auszahlung €87.29 bestätigt + Monatsübersicht
4
Widerruf / Dispute Handling
Käufer hat 14 Tage Widerrufsrecht (§355 BGB).
Bei Dispute: Stripe hält Transfer automatisch zurück (Stripe Connect Dispute Flow).
FYNDDS entscheidet manuell ob Payout freigegeben wird.

10 — Webhook-Architektur

Welche Webhooks wann feuern
der komplette Event-Flow

QuelleEventUnser EndpointWas passiert
Stripe payment_intent.succeeded /stripe-webhook Order erstellen, Asset = "sold", initiate-rerouting aufrufen
Stripe account.updated /stripe-webhook stripe_onboarding_completed in profiles setzen
Stripe charge.dispute.created /stripe-webhook Order flaggen, Admin-Alert, Transfer pausieren
DHL DELIVERED_TO_SENDER /receive-webhook Asset Status → "in_transit", Rerouting-Queue prüfen
DHL IN_TRANSIT /receive-webhook Rerouting-Queue prüfen: Käufer da? → sofort PATCH ausführen
DHL OUT_FOR_DELIVERY /receive-webhook Wenn Ziel = Käufer: warte auf DELIVERED. Wenn Ziel = Fulfillment: vorbereiten
DHL DELIVERED /tracking-webhook process-payout triggern, E-Mail senden, Order = "completed"
DHL EXCEPTION /receive-webhook Admin-Alert, Order auf "exception", Käufer benachrichtigen
DHL Webhook einrichten: Im DHL Developer Portal unter "Push Notifications" die FYNDDS-URL eintragen: https://sidjdgxnjcnornstjolt.supabase.co/functions/v1/receive-webhook. DHL sendet dann bei jedem Scan-Event einen POST mit dem Tracking-Status.
