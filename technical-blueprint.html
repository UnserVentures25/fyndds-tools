von Retoure bis Auszahlung

02 — Händler-Onboarding

Was ein Händler tun muss
einmalig, 10 Minuten

Jeder Händler braucht einen Stripe Connect Express Account. Das ist Pflicht — kein Stripe Connect, kein Listing. Die Einrichtung läuft vollautomatisch.

1
Händler registriert sich auf fyndds.com
E-Mail, Firmenname, USt-ID, Adresse. Supabase Auth erstellt den User. Tabelle profiles bekommt role = 'retailer'.
2
Stripe Connect Express Onboarding
Edge Function retailer-setup wird aufgerufen.
Stripe POST /v1/accounts → type: "express"
Stripe POST /v1/account_links → Onboarding-URL
Händler gibt bei Stripe Bankdaten + Ausweis an. Stripe verifiziert alles. FYNDDS muss nichts manuell prüfen.
3
API-Key für Shop-Integration
Händler bekommt einen FYNDDS API-Key. Damit kann der Shop-System (Shopify/WooCommerce) automatisch Retouren melden.
4
Fertig
✓ Stripe Connect aktiv → Geldfluss möglich
✓ DHL-Aufkleber werden automatisch durch FYNDDS erstellt
✓ Artikel erscheinen sofort auf dem Marketplace
03 — Return-Flow

Wie eine Retoure ins System kommt
drei Wege, ein Ergebnis

Eine Retoure kann auf drei Wegen bei FYNDDS landen. In allen Fällen erstellt FYNDDS den DHL-Aufkleber und listet den Artikel.

Weg A — API-Integration
Shop schickt bei jeder Retoure automatisch POST /api/returns mit Artikeldaten. Für Shopify / WooCommerce / Shopware.
Weg B — Dashboard
Händler lädt Artikel manuell im FYNDDS-Dashboard hoch. Foto, Beschreibung, Zustand, Mindestpreis.
Weg C — Bulk CSV
Händler schickt CSV mit vielen Retouren auf einmal. FYNDDS importiert alle, erstellt Labels und listet alles.

Rerouting technisch erklärt
exakt was passiert

FYNDDS erstellt alle Retourenaufkleber unter dem eigenen DHL-Geschäftskonto (EKP-Nummer). Das gibt FYNDDS die Autorität, Pakete während des Transports umzuleiten.

Schritt 1 — DHL Label erstellen
POST https://api-eu.dhl.com/parcel/de/v1/shipments

{
  "billingNumber": "FYNDDS_EKP_NUMMER02",
  "shipper": { "name1": "FYNDDS GmbH", ... },
  "consignee": { "name1": "Kundenname", "city": "Berlin", ... },
  "services": {
    "returnAddress": { "name1": "FYNDDS Fulfillment", ... }
  }
}

→ Response: { "shipmentTrackingNumber": "00340433836536510000", "label": {...} }
DHL Events — wann Rerouting möglich ist
DELIVERED_TO_SENDER — Übergabe an DHL durch Kunden Rerouting möglich
IN_TRANSIT — Paket im Netz Rerouting möglich
OUT_FOR_DELIVERY — Auf dem Zustellfahrzeug zu spät für Rerouting
DELIVERED — Zugestellt → Payout triggern
Schritt 2 — Adresse ändern (Das Herzstück)
// Nur möglich wenn Status = IN_TRANSIT oder DELIVERED_TO_SENDER

PATCH https://api-eu.dhl.com/parcel/de/v1/shipments/00340433836536510000/address

{ "consignee": { "name1": "Max Mustermann", "city": "München", "postalCode": "80331" } }

→ 200 OK: { "result": "address_changed" }
→ 409 Conflict: OUT_FOR_DELIVERY — Paket geht zu FYNDDS Fulfillment, wird von dort neu versendet
✓ Idealfall
Käufer da + Paket in Transit → Adresse ändern → direkt zum Käufer → Payout.
⚡ Fallback
Paket zu spät oder kein Käufer → zu FYNDDS Fulfillment → Listing bleibt aktiv → Versand aus Lager.
05 — Checkout & Zahlung

Wie der Käufer zahlt
Stripe Connect + on_behalf_of

Der Käufer zahlt einmal. Das Geld wird sofort aufgeteilt: Händler bekommt seinen Teil, FYNDDS behält Provision + Rerouting-Gebühr. Die Rechnung kommt vom Händler — nicht von FYNDDS.

Berechnung + Stripe Session
Berechnung:
   Produktsubtotal:       €89.99
   Rerouting-Gebühr:    + €2.99
   ────────────────────────────
   Gesamtbetrag Käufer:  €92.98

   FYNDDS Provision:     min(€15, max(€2.50, €89.99 × 3%)) = €2.70
   Händler bekommt:      €89.99 − €2.70 = €87.29
   FYNDDS behält:        €2.70 + €2.99 = €5.69

Stripe Session:
{
  payment_method_types: ["card", "sepa_debit", "link"],
  payment_intent_data: {
    on_behalf_of: "acct_händler_id",     // Rechnung kommt vom Händler
    transfer_data: { destination: "acct_händler_id", amount: 8729 },
    application_fee_amount: 270          // €2.70 an FYNDDS
  }
}

07 — Payout

Wie der Händler sein Geld bekommt
automatisch nach Zustellung

1
DHL Webhook: DELIVERED Event
DHL sendet POST mit Event DELIVERED an /receive-webhook. Edge Function findet zugehörige Order.
2
process-payout wird aufgerufen
Stripe POST /v1/transfers → €87.29 an Händler Connect Account
FYNDDS behält €5.69 (Provision + Rerouting) automatisch.
3
E-Mails via Resend
Käufer: Zustellbestätigung. Händler: Auszahlung €87.29 bestätigt.
4
Widerruf / Dispute
Käufer hat 14 Tage Widerrufsrecht. Bei Dispute: Stripe hält Transfer zurück. FYNDDS entscheidet manuell.

01 — Das Problem

Was gerade passiert
warum Retouren ein Milliarden-Problem sind

In Deutschland werden jedes Jahr 530 Millionen Pakete zurückgeschickt. Das klingt abstrakt — konkret bedeutet es: Ein Online-Shop mit 100 Bestellungen pro Monat hat durchschnittlich 20-30 Retouren. Jede kostet Geld.

Was passiert heute: Paket kommt zurück → liegt im Lager → verliert jeden Tag Wert → irgendwann wird es mit 50% Rabatt verkauft oder vernichtet. Zwischen Rückkunft und Wiederverkauf vergehen oft 4-8 Wochen. In dieser Zeit verliert der Artikel 20-40% seines Wertes.
Was FYNDDS macht: Das Paket kommt gar nicht erst ins Lager. Es geht direkt vom Rückwegfahrzeug zum nächsten Käufer. Der Händler spart Lagerkosten, Handling und Zeit. Der Käufer bekommt Markenware mit Rabatt. FYNDDS verdient an der Transaktion.
02 — Zwei Wege

Wie Händler zu FYNDDS kommen
mit oder ohne bestehendes System

Es gibt Händler die noch kein spezielles Return-Management-Tool nutzen, und Händler die bereits ein Tool wie 8returns verwenden. FYNDDS funktioniert für beide — ohne dass der Händler sein bestehendes System aufgeben muss.

Flow A — Händler kommt direkt zu FYNDDS
Der Händler hat kein Return-Management-System, oder er möchte wechseln. Er nutzt FYNDDS als komplette Lösung: Return-Portal, DHL-Label, Listing, Verkauf, Payout — alles aus einer Hand.

Vorteil: Maximales Rerouting möglich, weil FYNDDS von Anfang an alle Labels erstellt.
Für wen: Kleine bis mittlere Händler, Shopify/WooCommerce-Shops.
Flow B — Händler nutzt 8returns + FYNDDS parallel
Der Händler nutzt bereits ein Tool wie 8returns und will es nicht wechseln. FYNDDS hängt sich als Resale-Layer davor. Der Händler muss nur eine URL in 8returns eintragen.

Vorteil: Kein Systemwechsel, kein Risiko, sofort live.
Für wen: Händler die bereits Return-Software nutzen.
Was ist 8returns? 8returns (8returns.com) ist ein Return-Management-Tool für Online-Shops. Es bietet ein gebrandetes Return-Portal für Kunden, automatische Label-Generierung und ein Händler-Dashboard. Es löst die Logistik — aber nicht die Frage, wie man die Retoure wieder zu Geld macht. Genau da setzt FYNDDS an.
03 — Flow A

Händler nutzt alles über FYNDDS
der ideale Weg

FYNDDS übernimmt alles: das Return-Portal für den Endkunden, die DHL-Labels, den Marketplace und die Auszahlung. Der Händler hat keinen Mehraufwand — er bekommt einfach am Ende Geld.

1
Händler listet seine Retouren auf FYNDDS
Über das Dashboard oder die API meldet der Händler, welche Produkte als Retoure kommen werden. FYNDDS berechnet automatisch einen Verkaufspreis (z.B. 60% des Originalpreises) und veröffentlicht den Artikel im Shop.
2
Endkunde des Händlers will zurückschicken
Der Kunde des Händlers geht auf eine spezielle Rückgabe-Seite (kann unter der Domain des Händlers laufen, wird aber von FYNDDS betrieben). Er gibt seinen Retourengrund ein und bekommt automatisch einen DHL-Aufkleber per E-Mail zugeschickt.
!
Wichtig: FYNDDS erstellt den DHL-Aufkleber — nicht der Händler
Das ist der entscheidende Punkt. Der DHL-Aufkleber wird im Namen von FYNDDS erstellt (unter FYNDDS' eigenem DHL-Geschäftskonto). Dadurch ist FYNDDS rechtlich der "Absender" des Pakets — und hat die Befugnis, die Lieferadresse später per API zu ändern. Würde der Händler den Aufkleber selbst erstellen, wäre das nicht möglich.
3
Käufer findet den Artikel auf fyndds.com und kauft
Sobald ein Käufer zahlt: Stripe verarbeitet die Zahlung. Das Geld wird sofort in zwei Teile aufgeteilt — Händler-Anteil und FYNDDS-Anteil.
✓
FYNDDS leitet das Paket direkt zum Käufer um
Wenn das Paket noch unterwegs ist (der Händlers-Kunde hat es gerade abgegeben): FYNDDS ändert per DHL API die Lieferadresse. Das Paket fährt direkt zum Käufer. Der Händler sieht das Paket nie wieder. Kein Lager, kein Handling, kein Neuversand.
4
Paket zugestellt → Händler bekommt sein Geld
DHL meldet: Paket zugestellt. FYNDDS überweist automatisch den Händler-Anteil (Verkaufspreis minus Provision). Der Händler bekommt eine E-Mail mit Auszahlungsbestätigung.
04 — Flow B

Händler nutzt 8returns + FYNDDS
kein Systemwechsel nötig

Der Händler behält 8returns genau so wie es ist. Er trägt einmalig eine FYNDDS-URL in 8returns ein — und ab diesem Moment wird jede neue Retoure automatisch auf FYNDDS gelistet und verkauft, wenn möglich sogar umgeleitet.

Das Kernproblem bei dieser Integration: Normalerweise erstellt 8returns die DHL-Labels — mit dem Händler als Absender. Das bedeutet: 8returns (oder der Händler) kontrolliert das Paket, nicht FYNDDS. FYNDDS hat dann keine Befugnis, die Adresse zu ändern.

Die Lösung: 8returns hat eine "Custom Carrier" Funktion. Der Händler trägt dort FYNDDS als seinen Paketdienstleister ein. Ab dann erstellt FYNDDS alle Labels — unter FYNDDS' DHL-Konto. Volles Rerouting möglich.
Option 1 — Webhook-Kopplung (empfohlen)
8returns informiert FYNDDS sobald ein Return-Request eingeht. FYNDDS erstellt das Label.

✓ Händler trägt FYNDDS Webhook-URL in 8returns ein
✓ Bei jedem Return: 8returns → POST an FYNDDS
✓ FYNDDS erstellt DHL-Label (unter FYNDDS EKP)
✓ 8returns-Dashboard zeigt Tracking weiterhin
✓ Volle Rerouting-Befugnis für FYNDDS
Option 2 — 8returns Custom Carrier
Händler konfiguriert FYNDDS als "Custom Carrier" in 8returns — alle Labels kommen von FYNDDS.

✓ 8returns unterstützt Custom Carrier via API-Key
✓ Kein Webhook nötig — 8returns ruft FYNDDS direkt auf
✓ Transparenter für Händler (weniger Konfiguration)
✓ Einfachste Integration: nur eine Einstellung
✓ Volle Rerouting-Befugnis für FYNDDS
1
Einmalig: Händler trägt FYNDDS in 8returns ein
Einmalige Einstellung in 8returns: entweder als "Custom Carrier" (eine URL + API-Key) oder als Webhook-Empfänger. Das dauert 5 Minuten. Danach ändert sich der Workflow des Händlers nicht — 8returns sieht weiterhin alle Retouren im Dashboard.
2
Endkunde gibt Retoure in 8returns ein (wie gewohnt)
Für den Endkunden des Händlers ändert sich gar nichts. Er geht auf dieselbe Return-Seite wie bisher, gibt seinen Grund ein. 8returns verarbeitet den Request — und schickt dabei eine Anfrage an FYNDDS.
!
FYNDDS übernimmt Label-Erstellung + Listing
8returns schickt Produkt- und Kundendaten an FYNDDS. FYNDDS erstellt in Sekunden:
① DHL-Label unter eigenem EKP (Rerouting möglich)
② Produkt-Listing auf fyndds.com (sofort sichtbar für Käufer)
8returns bekommt Tracking-Nummer zurück und zeigt sie im Dashboard an.
3
Ab hier: identisch wie Flow A
Käufer findet Artikel auf FYNDDS → kauft → Zahlung aufgeteilt → Rerouting wenn möglich → Payout nach Zustellung. Der Händler merkt im Alltag keinen Unterschied — außer dass er am Ende mehr Geld hat.

05 — Warum wechseln / warum parallel nutzen?

Was FYNDDS besser macht
in einfachen Worten

8returns und FYNDDS lösen verschiedene Probleme. 8returns ist ein Logistik-Tool — es organisiert wie Pakete zurückkommen. FYNDDS ist ein Umsatz-Tool — es macht aus Retouren wieder Einnahmen.

8returns allein

  • Strukturierter Return-Prozess für Kunden
  • Automatische Label-Generierung
  • Übersicht über alle Retouren
  • Retoure bleibt ein Kostenpunkt
  • Artikel muss selbst weiterverkauft werden
  • Kein Rerouting — Paket muss ins Lager
  • Lagerkosten + Handling + Neuversand

8returns + FYNDDS

  • Alles von 8returns bleibt erhalten
  • Retoure wird automatisch zu Umsatz
  • 50%+ der Pakete kommen nie ins Lager
  • Käufer-Netzwerk von FYNDDS bereits vorhanden
  • Automatischer Payout nach Verkauf
  • Eine URL eintragen — fertig
  • Kein Risiko: FYNDDS verdient nur bei Verkauf
KriteriumNur 8returns8returns + FYNDDS
Retoure als Umsatz Retoure = Kostenpunkt. Nur Logistik-Optimierung. Retoure = Umsatz. Jede Retoure wird automatisch gelistet und verkauft.
Rerouting Nicht möglich. Paket muss immer ins Lager. 50%+ der Pakete kommen nie ins Lager. Spart €4–8 pro Paket (Handling + Lager).

Käufer-Netzwerk Händler muss selbst B-Ware vermarkten. FYNDDS bringt fertige Käufer. Keine eigenen Marketingkosten für B-Ware.

Jeder kritische Schritt im Detail

SCHRITT 4 — NICHT IMPLEMENTIERT
DHL Retourenlabel erstellen — create-dhl-label
Käufer A braucht ein DHL-Label zum Abschicken. Dieses Label muss zwingend unter dem FYNDDS EKP (Kundennummer) erstellt werden — sonst kann FYNDDS das Paket später nicht umleiten.

API-Aufruf:
POST https://api-eu.dhl.com/parcel/de/shipping/v2/orders
Body: billingNumber = FYNDDS_EKP · Shipper = Käufer-A-Adresse · Consignee = FYNDDS Depot
Antwort: shipmentNumber (Trackingnummer) + label (PDF Base64)

Danach: return_assets.tracking_number = shipmentNumber speichern → Label-PDF per Email an Käufer A.
⚠ Ohne diesen Schritt ist das gesamte Rerouting in Produktion unmöglich.
Wenn der Händler selbst ein DHL-Label erstellt hat → FYNDDS kann nicht umleiten → DHL antwortet mit 403 Forbidden.
SCHRITT 6 — DHL WEBHOOK
DHL: picked_up → Asset wird auf Marketplace gelistet
DHL scannt das Paket bei Abholung → sendet HTTP POST an /functions/v1/tracking-webhook.
Header: x-dhl-signature: HMAC-SHA256(body, DHL_WEBHOOK_SECRET)

Was FYNDDS macht:
1. Signatur validieren — wenn Secret gesetzt aber keine Signatur → 401 Reject
2. Asset per tracking_number suchen
3. status = "listed", sales_window_start = NOW(), sales_window_end = NOW() + 48h
4. Asset erscheint auf Marketplace — Käufer B kann jetzt kaufen
Corner Cases: Doppelter Webhook → idempotent (kein Problem). | Tracking-Number unbekannt → 200 OK (kein Retry-Storm). | 48h Sales-Window ist hardcoded — sollte pro Händler konfigurierbar sein. | Was passiert wenn Window abläuft und niemand kauft? → Asset braucht einen Rückgabe-Flow.
SCHRITT 9 — RACE CONDITION
create-checkout — zwei Käufer gleichzeitig
Käufer B klickt "Kaufen" → POST /functions/v1/create-checkout:
1. Prüft: status = "listed" AND sales_window_end > NOW()
2. Setzt status = "reserved"
3. Erstellt Stripe Checkout Session: on_behalf_of = händler_stripe_account_id, Platform Fee berechnet
4. Gibt checkoutUrl zurück → Redirect zu Stripe
⚠ Race Condition: Zwei Käufer klicken gleichzeitig "Kaufen". Beide sehen "listed", beide erstellen Sessions, beide zahlen → zwei Orders für ein Asset!
Fix: UPDATE return_assets SET status='reserved' WHERE id=X AND status='listed' — wenn 0 Rows updated → Asset weg, keine Session erstellen.
⚠ Abgebrochener Checkout: Asset bleibt dauerhaft auf "reserved" wenn Käufer B wegklickt. Nötig: Cron-Job der nach X Minuten "reserved" → "listed" zurücksetzt. Nicht implementiert.
SCHRITTE 12–13 — KERNFUNKTION
DHL Rerouting — das Herzstück von FYNDDS
Ablauf in initiate-rerouting:
1. DHL OAuth Token: POST /parcel/de/account/auth/ropc/v1/token
2. Status check: GET /track/shipments?trackingNumber={nr} → geblockt bei: out_for_delivery, delivered, returned, failure
3. Adresse ändern: PATCH /parcel/de/v1/shipments/{nr}/address
   Body: { consignee: { name1, addressStreet, postalCode, city, country } }
4. DHL 409 → Fallback: Order cancelled + Asset re-listed + kein automatischer Refund!
5. DHL 2xx → Rerouting bestätigt → DB schreiben → Email an Händler
⚠ EKP-Problem (kritischstes Problem): DHL erlaubt Rerouting NUR wenn das Label unter dem FYNDDS EKP erstellt wurde. Das ist der Grund warum Schritt 4 (create-dhl-label) die absolute Grundvoraussetzung ist.
⚠ Kein automatischer Stripe Refund bei Fehlschlag: Rerouting schlägt fehl → Order cancelled + re-listed — aber Käufer B hat bereits gezahlt und erhält kein automatisches Refund. Muss implementiert werden: stripe.refunds.create({ payment_intent: pi_XXX }).
⚠ out_for_delivery nicht geblockt: Wenn DHL "out_for_delivery" Webhook kommt, wird das Asset aktuell NICHT automatisch auf "unavailable" gesetzt. Käufer B kann noch kaufen, Rerouting schlägt dann mit 409 fehl → Refund-Problem.
SCHRITT 16 — AUSZAHLUNG
process-payout — Stripe Transfer an Händler
Getriggert durch DHL "delivered" Webhook → process-payout:
stripe.transfers.create({ amount: netCents, currency: "eur", destination: "acct_RETAILER" })

Berechnung: netCents = salePriceCents − platformFeeCents − 299 (€2,99 Rerouting Fee)
Platform Fee: derzeit hardcoded 3% — sollte tier-basiert aus pricingEngine.ts kommen.
⚠ Doppel-Auszahlung: DHL "delivered" Webhook kann mehrfach gesendet werden. Kein Idempotenz-Check → Stripe Transfer zweimal ausgeführt = doppelte Auszahlung an Händler.
Fix: Unique Constraint auf payouts.order_id + vor Transfer prüfen ob bereits ausgezahlt.
⚠ Inaktiver Stripe Account: Wenn Händler seinen Stripe-Account deaktiviert → Transfer schlägt fehl. Geld steckt im FYNDDS Platform Account. Nicht verloren, aber blockiert — manueller Eingriff nötig.
